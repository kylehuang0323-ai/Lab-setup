Step 1：基础设置

  • NAS 固定 IP：192.168.0.81
  • RAID 模式配置（SHR / RAID 5+）
  • DSM 更新到最新版本

Step 2：创建共享文件夹

路径：DSM → 控制面板 → 共享文件夹 → 新增

2.0 先配置本地群组（推荐）

权限分配建议以本地群组为主，不要逐个本地用户单独授权。这样后续新增/删除账号时，只需调整群组成员，不必逐个修改共享文件夹权限。

对象建议
本地群组创建 lab-admins、lab-users
本地用户使用实际登录 DSM 的账号；管理员加入 lab-admins，普通用户加入 lab-users
授权方式优先给群组授权，尽量避免给普通用户逐个单独授权

DSM 路径：控制面板 → 用户和群组 → 群组

  1. 新建群组 lab-admins（Lab 管理员）
  2. 新建群组 lab-users（Lab 普通用户）
  3. 将实际管理员账号加入 lab-admins
  4. 将普通使用账号加入 lab-users
💡
建议：如果 DSM 默认群组 users 已带权限，请同步检查并收紧，避免与后续自定义群组权限冲突。

2.1 创建 GoldenImages

配置项值
名称GoldenImages
描述Clonezilla 系统镜像存储
所在存储空间Volume 1
隐藏子文件夹✅ 勾选「对没有权限的用户隐藏」
回收站❌ 不勾选（镜像文件大，不需回收站）
权限lab-admins → 读写；lab-users → 禁止访问

2.2 创建 LabData

配置项值
名称LabData
描述用户工作数据（映射为 Z: 盘）
所在存储空间Volume 1
回收站✅ 勾选（保护用户误删数据）
权限lab-admins → 读写；lab-users → 读写

创建完后，进入 File Station → LabData，手动创建 12 个子文件夹：

# 位置：NAS SSH 终端 ssh <NAS管理员用户名>@192.168.0.81 for i in $(seq -w 1 12); do mkdir -p /volume1/LabData/WS$i done # 验证结果： ls /volume1/LabData/ # 应显示：WS01 WS02 WS03 ... WS12

2.3 创建 Software

配置项值
名称Software
描述驱动、安装包、License 文件仓库
权限lab-admins → 读写；lab-users → 只读

建议在 Software 下创建子目录：

# 位置：NAS SSH 终端 mkdir -p /volume1/Software/{Drivers,Tools,Licenses,ISOs} # Drivers/ — AMD芯片组、GPU、网卡驱动 # Tools/ — VS Code, Python, Git, 7-Zip 等安装包 # Licenses/ — 软件授权文件 # ISOs/ — Windows 安装镜像、Clonezilla Live ISO

2.4 创建 docker

配置项值
名称docker
描述Docker 容器持久化数据
权限lab-admins → 读写；lab-users → 禁止访问
隐藏✅ 对普通用户隐藏
💡
注意：ActiveBackup 文件夹不需要手动创建，安装 Active Backup for Business 套件时会自动生成。

Step 3：启用文件共享服务

3.1 启用 SMB 服务

路径：控制面板 → 文件服务 → SMB

配置项设置
启用 SMB 服务✅ 开启
最高 SMB 协议SMB3
最低 SMB 协议SMB2（兼容性）
工作组WORKGROUP
传输加密自动（或强制，按安全需求）
✅
验证方法：在任意 Windows 电脑上，按 Win+R 输入 \\192.168.0.81，应能看到共享文件夹列表。

3.2 启用 NFS 服务（供 Clonezilla 使用）

路径：控制面板 → 文件服务 → NFS

配置项设置
启用 NFS 服务✅ 开启
最高 NFS 协议NFSv4.1

然后配置 GoldenImages 的 NFS 权限：

共享文件夹 → GoldenImages → 编辑 → NFS 权限 → 新增

配置项值说明
服务器名称或 IP192.168.0.0/24允许整个内网段
权限读写Clonezilla 需要写入镜像
Squash映射所有用户为 admin避免权限问题
安全性sys标准 UNIX 认证
启用异步✅提升写入性能
允许非特权端口连接✅兼容 Clonezilla
⚠️
注意：NFS 仅对 GoldenImages 开放，LabData 使用 SMB 即可。不要对所有文件夹都开 NFS。
💡
使用边界：NFS 主要用于 Clonezilla Live / Linux 客户端 做镜像备份与恢复，不是给日常 Windows 工作站挂载使用。12 台 Windows 工作站平时应通过 SMB 访问 LabData 和 Software。

Step 4：安装必要套件

路径：DSM → 套件中心

4.1 必装套件

套件名称用途安装后操作
Active Backup for Business 工作站整机备份 & 裸金属恢复 安装后暂不配置，等工作站部署完再添加设备
Container Manager Docker 容器管理（原 Docker 套件） 安装后进入下一步部署服务

4.2 可选套件

套件名称用途建议

4.3 Active Backup 安装后确认

  • 套件中心显示「已安装」状态
  • 自动创建了 ActiveBackupforBusiness 共享文件夹
  • 可通过 DSM 主菜单打开 Active Backup 控制台
  • 暂不需要添加设备（等 Step 5 后再操作）

4.4 Container Manager 首次启用

重要：安装 Container Manager 后，不需要单独再安装 Docker。DSM 7.x 中，Docker 运行环境已经包含在 Container Manager 内。

项目建议
桥接网络配置向导保持默认，直接跳过/下一步；当前方案不需要自定义桥接网络
容器网络使用默认网络 + 端口映射即可
首次进入后确认左侧可见「容器 / 映像 / 项目 / 注册表」页面即可
是否要先拉镜像否，后续按 SSH 命令直接 docker run 即可
💡
说明：本方案中的 Uptime Kuma、RustDesk、Wiki.js 都采用默认 bridge 网络 + 端口映射，访问地址仍是 NAS 自身 IP，例如 http://192.168.0.81:3001。

4.5 开启 SSH（供后续命令行部署）

DSM 路径：控制面板 → 终端机和 SNMP → 终端机

  1. 勾选 启用 SSH 功能
  2. 端口保持默认 22
  3. 点击 应用

后续命令使用 NAS 实际管理员用户名 登录，而不是必须使用 admin。例如：

ssh <NAS管理员用户名>@192.168.0.81
⚠️
sudo 密码说明：后续看到命令行中的 Password: 提示时，输入的是当前 NAS 登录账号的密码，不是 Uptime Kuma 或 RustDesk 的密码。

Step 5：Docker 服务部署

5.1 准备工作

本节以下所有命令均在NAS SSH 终端中执行。

# 位置：NAS SSH 终端 ssh <NAS管理员用户名>@192.168.0.81 # 创建 Docker 数据目录 sudo mkdir -p /volume1/docker/uptime-kuma sudo mkdir -p /volume1/docker/rustdesk sudo mkdir -p /volume1/docker/wikijs # 验证 Docker 引擎可用 sudo docker ps

5.2 部署 Uptime Kuma（工作站状态监控）

# 位置：NAS SSH 终端 sudo docker run -d \ --name uptime-kuma \ --restart=always \ -p 3001:3001 \ -v /volume1/docker/uptime-kuma:/app/data \ louislam/uptime-kuma:1

首次配置：

  1. 浏览器访问 http://192.168.0.81:3001
  2. 设置管理员用户名和密码
  3. 如果打开后直接进入登录/初始化页面，说明容器启动成功
  4. 后续等工作站上线后，添加 12 个 Ping Monitor：
    • 类型：Ping
    • 名称：GMK-WS01 ~ GMK-WS12
    • 地址：192.168.0.101 ~ 192.168.0.112
    • 心跳间隔：60 秒
  5. 配置告警通知（可选 Webhook / 邮件 / 企业微信）

5.3 部署 RustDesk 中继服务器（远程管理）

# 位置：NAS SSH 终端 # RustDesk 需要两个服务：hbbs (ID注册) + hbbr (中继) sudo docker run -d \ --name rustdesk-hbbs \ --restart=always \ -p 21115:21115 -p 21116:21116 -p 21116:21116/udp -p 21118:21118 \ -v /volume1/docker/rustdesk:/root \ rustdesk/rustdesk-server hbbs sudo docker run -d \ --name rustdesk-hbbr \ --restart=always \ -p 21117:21117 -p 21119:21119 \ -v /volume1/docker/rustdesk:/root \ rustdesk/rustdesk-server hbbr

部署后验证：

# 位置：NAS SSH 终端 sudo docker ps # 取 RustDesk 公钥（推荐直接从宿主机挂载目录读取） sudo cat /volume1/docker/rustdesk/id_ed25519.pub

如果 sudo docker exec rustdesk-hbbs cat ... 提示找不到 cat，属于容器镜像内命令缺失，可直接读取宿主机挂载目录中的 id_ed25519.pub 文件。

💡
RustDesk 客户端配置（在每台工作站上）：
1. 安装 RustDesk 客户端
2. 进入设置 → ID/Relay Server / Network
3. 填写：ID Server = 192.168.0.81、Relay Server = 192.168.0.81
4. 将上一步命令输出的公钥内容填入 Key
5. 保存后客户端应显示「Ready / 就绪」

5.4 部署 Wiki.js（内部文档站，可选）

# 位置：NAS SSH 终端 sudo docker run -d \ --name wikijs \ --restart=always \ -p 3000:3000 \ -v /volume1/docker/wikijs:/wiki/data/content \ -e DB_TYPE=sqlite \ -e DB_FILEPATH=/wiki/data/content/wiki.sqlite \ ghcr.io/requarks/wiki:2

访问 http://192.168.0.81:3000 完成初始化，用于存放：

  • 用户使用手册（数据保存策略、重启须知）
  • 管理员 SOP（月度维护、应急处理）
  • Lab 规章制度

Windows 工作站当前应该做什么

如果你的 12 台设备当前主要是 Windows 工作站，下一步重点不是给它们挂 NFS，而是先把 SMB、RustDesk、软件仓库、Golden Image 这条主线跑通。

  1. 选 1 台工作站作为标准机：先完成 Windows、驱动、常用软件、RustDesk、网络盘映射配置
  2. 访问 NAS SMB 共享：\\192.168.0.81，验证 LabData 可读写、Software 只读、docker 不可访问
  3. 映射网络盘：例如在 Windows 中执行 net use Z: \\192.168.0.81\LabData
  4. 配置 RustDesk 客户端：填写 ID Server = 192.168.0.81、Relay Server = 192.168.0.81、Key = RustDesk 公钥
  5. 从 Software 安装驱动和工具：AMD 驱动、VS Code、Python、Git、7-Zip、RustDesk 等
  6. 完成标准机后再做镜像：使用 Sysprep + Clonezilla 将标准机备份到 GoldenImages，再批量恢复到其余 11 台
🎯
建议顺序：不要一开始就逐台手工配置 12 台 Windows 机器。先完成 1 台标准机，再镜像复制，整体效率最高，也最符合这套 Lab 部署方案。
🎯
建议：NAS 验证完毕后，先把所有驱动和软件安装包上传到 \\192.168.0.81\Software，这样制作 Golden Image 时可以直接从 NAS 拉取，不依赖外网。
