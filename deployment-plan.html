🏗️ 一、整体架构

分层防护模型

L4 用户数据层 所有用户文件存 NAS / OneDrive，C 盘不留数据 NAS SMB + 文件夹重定向
L3 日常保护层 每次重启自动还原到基线状态，抵御误操作/病毒 Reboot Restore Rx Pro
L2 灾难恢复层 整机裸金属备份，硬盘故障可完全恢复 Synology Active Backup
L1 标准镜像层 Golden Image 模板，新机/重置时批量部署 Clonezilla + NAS 存储
L0 硬件基础层 12× GMK EVO-X2 + NAS + 千兆交换机 物理设备

🔄 二、NAS 替代 Proxmox 方案

🎯 核心决策：不保留 Proxmox 节点，NAS 承担所有管理职能

原因：12 台全部给用户使用 → 不浪费硬件资源 → NAS 已具备所有必要能力

原 Proxmox 功能 NAS 替代方案 实现方式 状态
镜像存储 & 批量部署 NAS 共享 + Clonezilla NFS/SMB 共享 Golden Image，Clonezilla 通过网络还原 ✅ 原生支持
整机备份 & 裸金属恢复 Synology Active Backup for Business 免费套件，Agent 部署到每台 Windows，自动增量备份 ✅ 原生支持
网络监控 Dashboard Docker: Uptime Kuma Synology Docker 跑轻量监控，Ping/HTTP 检查 12 台状态 🐳 Docker
内部文档 / Wiki Docker: Wiki.js 或 Web Station 管理员 SOP、用户手册托管在 NAS 上 🐳 Docker
远程唤醒 (WoL) NAS 脚本 / Uptime Kuma Scheduled Task 发送 Magic Packet 批量开机 ✅ 原生支持
PXE 网络启动 不需要 (Clonezilla USB 足够) 12 台规模 USB 并行部署 < 2h，PXE 过度工程 ⏸ 不需要
开发/测试 VM Synology VMM (如需要) NAS 如有足够内存可跑轻量 Linux VM 📦 可选
💡
结论：Synology NAS 完全覆盖 Proxmox 的管理功能， 且无需维护额外的虚拟化平台。Active Backup + Docker + 共享存储 = 完整的 Lab 管理中枢。

# BIOS 设置（已验证的 GMK AMI BIOS）： # • Advanced → CPU Config → SVM Mode: Enabled（备用，不影响裸金属） # • Advanced → CPU Config → IOMMU: Auto（裸金属无需 Enabled） # • Advanced → GFX Configuration → iGPU: Auto（裸金属使用默认） # • Security → Secure Boot: Enabled（裸金属建议开启） # • Boot → Boot Option #1: USB（安装时） # 安装 Windows 11： # 1. ⚠️ 七、容易忽略的问题

1
Windows License
12 台 Win11 需要授权。确认是否通过 Microsoft 365 E3/E5 获得 Windows Enterprise 权益， 还是需要单独购买 OEM/Volume License。若用 IoT Enterprise 则可免费用 UWF。
2
AMD Strix Halo 驱动兼容性
全新平台，Windows 自带驱动可能不完整。必须提前准备离线驱动包： 芯片组、GPU (Radeon 8050S)、网卡、USB 控制器。建议在 Golden Image 中验证所有设备管理器无感叹号。
3
Sysprep 与回滚软件冲突
Reboot Restore Rx Pro 必须在 Sysprep 之后、首次 OOBE 完成后再安装。 否则封装会破坏回滚基线。正确顺序：Sysprep → 克隆 → 部署 → OOBE → 装回滚软件 → 设基线。
4
回滚后 NAS 映射丢失
如果用户 Profile 被还原，net use persistent 映射可能丢失。 解决：用启动脚本（Task Scheduler 或 GPO Logon Script）每次登录自动映射。
5
时钟同步 & Hostname 冲突
12 台从同一镜像克隆后，如果 SID 未正确一般化，Windows 网络发现会冲突。 Sysprep /generalize 是关键步骤，不可跳过。
6
NAS 单点故障
如果 NAS 挂了，用户数据和备份都不可用。建议： • NAS 配置 RAID（SHR/RAID 5 以上） • 重要数据启用 Hyper Backup 到外部硬盘或云（Azure Blob 等）
7
12 台差异化需求
如果每台工作站有不同的软件/配置需求，需要： • 方案 A：一个 Golden Image + 部署后脚本差异化 • 方案 B：多个 Golden Image（维护成本高） • 建议用方案 A，post-deploy 脚本按工位号安装特定软件
8
远程管理工具
没有 Proxmox 后，日常管理 12 台需要远程方案： • RustDesk（开源远程桌面，自建中继服务器在 NAS Docker） • 或 PowerShell Remoting (WinRM) 批量执行命令 • 建议两者都装入 Golden Image
