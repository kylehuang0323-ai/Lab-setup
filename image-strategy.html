一、核心概念：源镜像 vs 运维基线

I-Lab 的镜像体系由两个不同层次组成，二者经常被混淆：

📀 Golden Image（源镜像）

  • Sysprep 泛化后的部署模板
  • 12 台共用同一份
  • 不含 hostname、IP、激活状态
  • 作用：批量克隆的起点
  • 存放位置：NAS /GoldenImages
  • 更新频率：重大变更时才更新

📸 Baseline Snapshot（运维基线）

  • 每台机器独立的日常回滚目标
  • 12 台各不相同（各有唯一身份）
  • 包含 hostname、IP、激活、回滚软件
  • 作用：每日回滚的恢复点
  • 存放位置：本地磁盘 / Reboot Restore
  • 更新频率：每月维护日重新设置
⚠️
关键区分：Golden Image 是"部署工件"，不是"运营基线"。
每日回滚恢复到的是 Baseline Snapshot（部署后拍摄），而不是重新从 Golden Image 克隆。
如果每次都从 Golden Image 重新部署，将会重复消耗 Windows 激活、丢失机器身份、增加配置时间。

三、Windows 激活保护逻辑

3.1 为什么必须在 OOBE 之后拍快照？

❌ 错误做法：每日回滚到 Sysprep 状态 Golden Image → 每次重启都重新 OOBE → 每次消耗一次激活 → 激活耗尽！ ✅ 正确做法：每日回滚到已激活的 Baseline Golden Image → 克隆 → OOBE(一次) → 激活(一次) → 📸快照 → 每日回滚到此 激活状态保存在快照中 → 永远不会重复消耗

3.2 不同授权方式的影响

授权方式激活次数限制回滚影响建议
OEM (主板绑定) 无限制（绑定硬件） ✅ 回滚后自动重新激活 GMK 预装的即此类型，最理想
KMS (批量激活) 无限制（网络验证） ✅ 只要 KMS 服务器可达即可 企业环境可选
MAK (多次激活密钥) 有限次数 ⚠️ 重新部署会消耗次数 务必在激活后拍快照
评估版 License 90天期限 ⚠️ 回滚不重置计时器 仅用于短期测试

3.3 Office 激活特别警告

🚨
Office 2024 离线密钥限制：每个密钥通常只有 25 次激活上限。

如果在 Golden Image 中预装 Office 但不在 Baseline Snapshot 中保留激活状态：
• 12 台机器 × 每次重新部署 = 快速耗尽激活次数
• 解决方案：在每台完成 Office 激活后再拍 Baseline Snapshot

Microsoft 365 在线激活尚未在此镜像模式中验证，建议优先使用 OEM 预装或 KMS。

五、月度维护 Re-baseline 完整流程

每月第 1 个周日执行，目的是更新系统补丁和软件后重新设置各台的 Baseline。

5.1 流程步骤

# ═══════════════════════════════════════════════ # 月度 Re-baseline 流程 (管理员操作) # ═══════════════════════════════════════════════ # Step 1: 解冻所有工作站 (通过 Reboot Restore Console 或远程) # 操作：Reboot Restore Console → 选择所有设备 → Disable Protection # 或远程逐台执行： Invoke-Command -ComputerName GMK-WS01,GMK-WS02,...,GMK-WS12 -ScriptBlock { # Reboot Restore CLI 解冻（具体命令视软件版本） & "C:\Program Files\Reboot Restore Rx Pro\RRCmd.exe" /disable } # Step 2: 批量执行更新 Invoke-Command -ComputerName GMK-WS01,...,GMK-WS12 -ScriptBlock { # Windows Update Install-Module PSWindowsUpdate -Force -Confirm:$false Get-WindowsUpdate -Install -AcceptAll -AutoReboot:$false # 软件更新 (winget) winget upgrade --all --accept-package-agreements --accept-source-agreements } # Step 3: 验证所有机器正常 Invoke-Command -ComputerName GMK-WS01,...,GMK-WS12 -ScriptBlock { # 检查设备管理器异常 Get-PnpDevice -Status Error,Degraded,Unknown # 检查 Windows 激活 cscript //nologo C:\Windows\System32\slmgr.vbs /xpr } # Step 4: 重新设置 Baseline Invoke-Command -ComputerName GMK-WS01,...,GMK-WS12 -ScriptBlock { # 重新冻结并设置新基线 & "C:\Program Files\Reboot Restore Rx Pro\RRCmd.exe" /enable /setbaseline } # Step 5: 更新 Golden Image (如有重大变更) # 选择 WS01 作为参考机： # 1. 解冻 WS01 # 2. Sysprep /generalize /oobe /shutdown # 3. Clonezilla 备份到 NAS (新版本号) # 4. 重新克隆 WS01 → OOBE → 设基线（恢复其独立身份） # Step 6: 触发 Active Backup 完整备份 # NAS → Active Backup → 手动触发全量备份 # Step 7: 更新 CHANGELOG # 记录本次更新内容到 NAS:/GoldenImages/CHANGELOG.md

5.2 Re-baseline 决策树

月度维护日到了 │ ├── 仅小补丁/安全更新？ │ → 解冻 → 更新 → 重设各台 Baseline → 完成 │ → 不需要更新 Golden Image │ ├── 有新软件/大版本变更？ │ → 解冻 → 更新 → 验证 → 重设 Baseline │ → 同时选一台 Sysprep → 更新 Golden Image 到 NAS │ → (Golden Image 仅用于未来新机/硬盘更换场景) │ └── 有硬件更换/新机器加入？ → 从 NAS Golden Image 克隆到新机 → OOBE → 分配 hostname/IP → 激活 → 设 Baseline → 无需影响其他 11 台

5.3 注意事项

⚠️
Re-baseline 不等于重新部署！
• Re-baseline = 在当前状态上更新基线快照点（保留机器身份）
• 重新部署 = 从 Golden Image 重新克隆（丢失机器身份，需重新走 OOBE）

正常月度维护只需 Re-baseline，不需要重新部署。
重新部署仅在硬盘故障、新机器加入、或镜像严重损坏时才需要。

六、异常场景处理

场景处理方式是否需要重新 OOBE
用户搞坏了系统 重启即恢复（Reboot Restore） ❌ 不需要
Reboot Restore 失效 Active Backup 裸金属恢复到最近备份 ❌ 不需要（备份含完整身份）
硬盘物理损坏 换盘 → Clonezilla 从 Golden Image 克隆 ✅ 需要（重走 OOBE + post-deploy）
批量系统崩溃 所有机器从 Golden Image 重新克隆 ✅ 需要（全部重走）
Golden Image 本身有问题 用 NAS 上的历史版本回退 ✅ 需要（从旧版本重新克隆）
新机器加入 Golden Image 克隆 → 正常流程 ✅ 需要（新机器走完整流程）
💡
恢复层级优先级：
① 重启还原（秒级，日常自愈） → ② Active Backup 恢复（分钟级，含完整身份） → ③ Golden Image 重新克隆（小时级，需要重新配置身份）
